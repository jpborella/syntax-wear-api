# Instruções para o Syntax Wear API

## Arquitetura e Padrões

### Estrutura MVC
Mantenha a separação clara:
- **Controllers**: Lidam com requisições HTTP, validação de input com Zod, e delegam lógica para services
- **Services**: Contêm toda a lógica de negócio, lançam erros simples (não formatados)
- **Middlewares**: Tratamento centralizado de autenticação e erros
- **Types**: Interfaces TypeScript centralizadas em `types/index.ts`
- **Utils**: Validadores Zod, instância Prisma, funções auxiliares

### Fluxo de Requisição
1. **Controller** valida input usando schema Zod
2. **Service** executa lógica de negócio
3. **Error Handler** middleware captura erros e formata respostas
4. **Response** é enviada ao cliente com metadados apropriados

## Validação e Tipos

### Schemas Zod
- Mantenha todos os schemas Zod em `utils/validator.ts`
- Use mensagens de erro em **português brasileiro**
- Aplique `.trim()` em strings, `.min()`, `.max()` conforme necessário
- Use `.optional()` para campos opcionais e `.nullable()` quando permitir null

**Exemplo:**
```typescript
export const productSchema = z.object({
    name: z.string().trim().min(2, "Nome deve ter no mínimo 2 caracteres."),
    price: z.number().positive("Preço deve ser maior que zero."),
    description: z.string().optional(),
    categoryId: z.number().int().positive(),
});
```

### Tipos TypeScript
- Centralize interfaces em `types/index.ts`
- Use **PascalCase** para nomes de tipos e interfaces
- Use `Partial<T>` para operações de atualização
- Documente tipos complexos com comentários JSDoc

**Exemplo:**
```typescript
export interface CreateProductRequest {
    name: string;
    price: number;
    description?: string;
    categoryId: number;
}

export interface UpdateProductRequest extends Partial<CreateProductRequest> {}
```

## Controllers

### Padrão de Controller
1. Valide input com Zod
2. Chame service correspondente
3. Retorne resposta com status apropriado
4. Deixe o middleware de erro tratar exceções

**Exemplo:**
```typescript
export const createProduct = async (request: FastifyRequest<{Body: CreateProductRequest}>, reply: FastifyReply) => {
    const validation = productSchema.parse(request.body);
    const product = await productService.create(validation);
    return reply.status(201).send({
        message: "Produto criado com sucesso.",
        data: product,
    });
};
```

### Nomeação de Funções de Controller
Use verbos descritivos que reflitam a ação:
- `create` / `createNew` (POST)
- `list` / `getList` (GET com múltiplos)
- `getById` / `get` (GET com ID)
- `update` / `updateExisting` (PATCH)
- `delete` / `deleteExisting` (DELETE)

## Services

### Padrão de Service
- Execute lógica de negócio pura
- Valide regras de negócio (duplicatas, referências, etc)
- Lance erros simples: `throw new Error("Mensagem descritiva")`
- Retorne dados sem formatação (o controller cuida do formato)

**Exemplo:**
```typescript
export const createProduct = async (payload: CreateProductRequest) => {
    const slugified = slugify(payload.name, { locale: 'pt' });
    
    // Valide regras de negócio
    const existingProduct = await prisma.product.findUnique({
        where: { slug: slugified },
    });
    
    if (existingProduct) {
        throw new Error("Produto com este nome já existe.");
    }
    
    const product = await prisma.product.create({
        data: { ...payload, slug: slugified },
    });
    
    return product;
};
```

### Queries Prisma Otimizadas
- Use `Promise.all()` para operações paralelas
- Implemente paginação com `skip` e `take`
- Retorne contagem total junto com dados paginados
- Use busca case-insensitive quando apropriado

**Exemplo:**
```typescript
export const listProducts = async (filters: ProductFilters) => {
    const where: any = {};
    
    if (filters.search) {
        where.OR = [
            { name: { contains: filters.search, mode: 'insensitive' } },
            { description: { contains: filters.search, mode: 'insensitive' } },
        ];
    }
    
    const skip = (Number(filters.page) - 1) * Number(filters.limit);
    const [products, total] = await Promise.all([
        prisma.product.findMany({ where, skip, take: filters.limit }),
        prisma.product.count({ where }),
    ]);
    
    return {
        data: products,
        total,
        page: filters.page,
        totalPages: Math.ceil(total / filters.limit),
    };
};
```

## Tratamento de Erros

### No Service
Sempre lance erros simples com mensagens descritivas em **português**:
```typescript
throw new Error("Usuário não encontrado.");
throw new Error("Email já está registrado.");
throw new Error("Permissão negada para acessar este recurso.");
```

### No Middleware de Erro
O middleware `error.middleware.ts` captura e formata:
- Erros de validação Zod (status 400)
- Erros de autenticação JWT (status 401)
- Erros genéricos (status 500)

**Resposta padrão:**
```typescript
{
    "message": "Descrição do erro",
    "errors": {} // Apenas em erros de validação
}
```

## Autenticação e Autorização

### Proteção de Rotas
Use o middleware `authenticate` para rotas protegidas:

```typescript
export default async function userRoutes(fastify: FastifyInstance) {
    // Todas as rotas abaixo precisam de autenticação
    fastify.addHook("onRequest", authenticate);
    
    fastify.get("/me", getCurrentUser);
    fastify.patch("/", updateProfile);
}
```

### Geração de JWT
```typescript
const token = request.server.jwt.sign(
    { userId: user.id, email: user.email },
    { expiresIn: "24h" }
);
```

### Verificação de Permissões
Implemente em services, não em controllers:
```typescript
export const deleteProduct = async (productId: number, requestingUserId: number) => {
    const product = await prisma.product.findUnique({ where: { id: productId } });
    
    if (!product) throw new Error("Produto não encontrado.");
    if (product.createdById !== requestingUserId) {
        throw new Error("Permissão negada.");
    }
    
    return prisma.product.delete({ where: { id: productId } });
};
```

## Padrões de Dados

### Soft Delete
Use `active: false` em vez de deletar registros fisicamente:
```typescript
// Ao invés de delete
await prisma.product.delete({ where: { id: productId } });

// Faça
await prisma.product.update({
    where: { id: productId },
    data: { active: false },
});

// E sempre filtre por active: true nas queries
where: { active: true }
```

### Validação de Unicidade
Sempre valide campos únicos antes de inserir/atualizar:
```typescript
const existingUser = await prisma.user.findUnique({
    where: { email: payload.email },
});

if (existingUser) throw new Error("Email já registrado.");
```

### Slugs
- Gere automaticamente a partir do nome usando `slugify()` com locale PT
- Valide se o slug já existe (garantir unicidade)
- Use em URLs, nunca em IDs

```typescript
import slugify from 'slugify';

const slug = slugify(name, { locale: 'pt', lower: true });
```

## Convenções de Nomeação

### Arquivos
- **Controllers**: `<resource>.controller.ts` (ex: `products.controller.ts`)
- **Services**: `<resource>.services.ts` (ex: `product.services.ts`) - Note: singular no serviço
- **Routes**: `<resource>.routes.ts` (ex: `products.routes.ts`)
- **Middlewares**: `<name>.middleware.ts` (ex: `auth.middleware.ts`)

### Código
- **Variáveis e funções**: `camelCase`
- **Constantes e tipos**: `PascalCase` ou `UPPER_CASE` para constantes
- **Booleanos**: Prefixo `is`, `has`, `should` (ex: `isActive`, `hasPermission`)

**Exemplo:**
```typescript
interface UserResponse { ... }
const DEFAULT_PAGE_SIZE = 10;
const isUserAdmin = user.role === 'admin';
const hasValidToken = !!request.jwtPayload;
```

## Respostas HTTP

### Estrutura Padrão
```typescript
{
    "message": "Descrição amigável em português",
    "data": { /* dados do recurso */ }  // Opcional, apenas em sucesso
}
```

### Status Codes
- **201**: Recurso criado (POST)
- **200**: Sucesso geral (GET, PATCH)
- **204**: Sucesso sem conteúdo (DELETE pode usar)
- **400**: Erro de validação (Zod)
- **401**: Não autenticado
- **403**: Sem permissão
- **404**: Recurso não encontrado
- **409**: Conflito (email duplicado, slug já existe)
- **500**: Erro interno do servidor

### Exemplos
```typescript
// Sucesso ao criar
return reply.status(201).send({
    message: "Produto criado com sucesso.",
    data: product,
});

// Sucesso ao listar
return reply.status(200).send({
    message: "Produtos listados com sucesso.",
    data: products.data,
    pagination: {
        total: products.total,
        page: products.page,
        totalPages: products.totalPages,
    },
});

// Erro (deixe o middleware tratar)
throw new Error("Produto não encontrado.");
```

## Boas Práticas Gerais

1. **Mensagens em Português**: Todas as mensagens de erro, validação e sucesso devem estar em português brasileiro
2. **Async/Await**: Use async/await, evite `.then().catch()`
3. **Error Handling**: Deixe o middleware centralizado tratar erros
4. **Type Safety**: Use TypeScript strict, evite `any`
5. **Validação em Camadas**: Controller valida formato, Service valida regras de negócio
6. **DRY**: Reutilize schemas Zod, interfaces e funções auxiliares
7. **Testes**: Mantenha funções services puras para facilitar testes
8. **Logging**: Use console ou logger estruturado para debug
9. **Performance**: Use `Promise.all()` para operações paralelas
10. **Segurança**: Nunca retorne senhas, tokens ou dados sensíveis
